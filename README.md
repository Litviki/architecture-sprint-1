# Задание 2:

В исходной монолитной системе управления торговыми площадками были выделены 9 доменов. Для каждого домена был сформирован микросервис.

1) Управление пользователями. 
Служит для хранения данных пользователей и формирования отчётов об их активности.

2) Авторизация пользователей. 
В задачи микросервиса входит формирование JWT токена в процессе аутентификации 
и валидация JWT токена в операциях, которые требуют зарегистрированных пользователей.

3) Услуги
Служит для добавления и редактирования предоставляемых услуг. Формирует списки услуг для операций фильтрации и поиска.

4) Заказы
Служит для создания и управления заказами. Позволяет получить список и статистику заказов для пользователя.

5) Товары
Порзволяет продавцу добавить и отредактировать товар, выставленный на аукцион, а покупателям ознакомиться со списком имеющихся товаров с возможностью фильтрации.

6) Аукционы
Позволяет продавцам открывать аукцион по добавленному товару, а покупателям присоединяться к покумне, устанавливать и менять ставку. 
В случае несогласия с результатами аукциона пользователь может подать апелляцию.
Администратор площадки может валидировать заявку и редактировать аукцион.

7) Платежи
Позволяет выполнять платежи с помощью внешних сервисов.

8) Техподдержка
Служит для сбора обратной связи от пользователей для оперативного решения проблем. Не требует аутентификации пользователей.

9) Оповещения
Служит для рассылки оповещений пользователям по электронной почте или иными доступными способами.

Подавляющее число операций являются синхронными. Операции оповещения пользователя не требуют синхронного
взаимодействия и могут выполняться через брокера сообщений (например, RabbitMQ).

Сервис авторизации используется практически во всех запросах и может стать узким местом системы. 
Применение масштабирования с помощью Kubernetes позволит решить эту проблему.

Платежные операции в данной системе выполняются в фоновом режиме - по окончании аукциона или согласования 
сторон при заказе услуги. Процесс взаимодействия заказчика и исполнителя услуг является многоступенчатым, 
и требует участия обеих сторон. Для обеспечения непротиворечивой работы системы на этапах оформления 
заказа услуги и проведения аукциона можно использовать паттерн Saga с оркестрацией. Оркестрация необходима 
потому, что некоторые этапы требуют ручного подтверждения со стороны пользователей.

Для повышения удобства разработки и улучшения UX, монолитный фронтенд был разделён на микрофронтенды:
* Заказ услуг
* Управление услугами
* Регистрация
* Аутентификация
* Личный кабинет пользователя
* Управление товарами
* Панель продавца аукциона
* Панель покупателя аукциона
* Форма обратной связи

Все микрофронтенды имеют гибридную композицию.
